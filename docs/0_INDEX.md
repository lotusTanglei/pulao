# Pulao 文档维护规范与索引 (Docs Convention)

## 📌 1. 文档结构公约 (The 5+1 Rule)

为了避免文档膨胀、信息过期和浪费 AI Token，本项目的 `docs/` 目录严格执行 **5+1 精简结构**。**除以下 6 个文件外，禁止在 `docs/` 根目录创建其他 `.md` 文件**。

| 文件名 | 职责与内容 | 更新时机 |
| :--- | :--- | :--- |
| `0_INDEX.md` | (本文件) 目录索引与维护规范 | 仅当重构文档体系时 |
| `1_ARCHITECTURE.md` | 核心模块映射、数据流向、关键架构决策(ADR) | 发生重大架构变更时 (如引入新框架) |
| `2_CAPABILITIES.md` | 当前系统**真实可用**的功能清单 (Checkbox 形式) | **完成一个新 Feature 提交时** |
| `3_ROADMAP.md` | 待办任务、未来三个阶段的发展规划 | 每隔 1-2 个月，或规划方向大改时 |
| `4_TEST_PLAN.md` | 核心功能的测试用例与验证标准 | 新增功能需提供配套验证方案时 |
| `USER_NOTEPAD.md` | 用户的私人草稿本，AI **禁止**主动修改 | 用户随心所欲 |

## ⚙️ 2. 文档更新机制 (制度化)

文档的价值在于**准确**而非长篇大论。为了节省 Token 并保持高效，执行以下制度：

1. **功能完成即同步 (Sync on Feature Complete)**：
   * 每次你要求 AI 开发完一个新功能（如：写完了 `ops_diagnostics.py`），在最终提交代码（Commit）前，必须附带指令：`更新 2_CAPABILITIES.md 和 3_ROADMAP.md`。
   * AI 会在 `2_CAPABILITIES.md` 中打钩，并从 `3_ROADMAP.md` 的待办中删掉该项。
2. **季检 (Quarterly Review)**：
   * 每隔一段时间（如重新接手项目时），让 AI 阅读这 5 个文件，判断内容是否与代码现状脱节。
3. **禁止长篇大论 (Token Saving Strategy)**：
   * **代码即文档**：具体的函数参数、实现细节必须写在 Python 代码的 Docstring 中。
   * **Markdown 只记宏观**：文档里只写“架构图”、“流程步骤”、“清单”，坚决**不贴大段源码**。

## 🗑️ 3. 历史垃圾桶
旧的、被推翻的文档已被移入 `archive/` 目录。日常开发时，AI 搜索工具应**主动忽略** `archive/` 目录，防止获取过时的错误上下文。
