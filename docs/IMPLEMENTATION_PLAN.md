# Pulao 功能实现计划

## 项目概述

Pulao 是一个基于 LangGraph 的智能运维 Agent，通过自然语言完成 Docker 部署、系统运维、K8s 集群管理及 DevOps 流程编排。

**核心功能模块：**

1. 单节点服务器运维
2. 创建服务器集群
3. 基于服务器集群进行 K8s 运维（接入已有环境）
4. DevOps 流程

---

## 阶段划分总览

| 阶段 | 功能范围 | 优先级 | 建议周期 |
|------|----------|--------|----------|
| 近期 (Phase 1) | 单节点运维增强 | P0 | 1-2 个月 |
| 中期 (Phase 2) | 集群管理 + K8s 运维接入 | P1 | 2-3 个月 |
| 远期 (Phase 3) | DevOps 与 GitOps 集成 | P2 | 3-6 个月 |

---

## 近期规划 (Phase 1)

### 目标
完善单节点运维能力，稳定现有 Agent 架构，建立安全操作机制。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 单节点 Docker 运维 | 软件安装、服务部署、Docker Compose 部署 | P0 |
| 运维监控 | 日志查看、资源监控、故障诊断 | P0 |
| 安全确认机制 | 危险操作人工确认 (Human-in-the-Loop) | P0 |
| 操作回滚 | 部署失败自动回滚 | P1 |
| 模板库管理 | Docker Compose 模板的增删改查 | P1 |

### 关键里程碑

```
里程碑 1.1: 完成危险操作确认机制
  - 定义危险操作列表（rm, drop, shutdown 等）
  - 实现操作前置确认流程
  - 完成时间: 1 周

里程碑 1.2: 实现部署回滚能力
  - 备份当前部署状态
  - 部署失败自动触发回滚
  - 完成时间: 2 周

里程碑 1.3: 完善模板库
  - 模板的 CRUD 操作
  - 模板分类和搜索
  - 完成时间: 2 周

里程碑 1.4: 增强监控诊断
  - 日志分析
  - 资源告警
  - 完成时间: 2 周
```

### 预期成果

- 单节点运维场景可投入使用
- 危险操作 100% 经过人工确认
- 部署成功率 > 95%，失败回滚率 100%

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 1-2 人 | 兼职/全职 |
| 测试服务器 | 1-2 台 | 用于测试部署操作 |
| AI API 额度 | 足够测试使用 | DeepSeek/OpenAI |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| Agent 误判执行危险操作 | 中 | 高 | 强制 Human-in-the-Loop，关键操作设多级确认 |
| AI API 调用失败 | 低 | 中 | 实现重试机制，优雅降级为手动模式 |
| Docker 操作导致系统不稳定 | 中 | 中 | 测试环境先行验证，非破坏性操作优先 |

## 中期规划 (Phase 2)

### 目标
扩展到多节点集群管理，实现 K8s 运维能力。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 集群定义与管理 | 多台服务器的逻辑分组 | P0 |
| SSH 批量操作 | 并行/串行执行命令 | P0 |
| 集群状态同步 | 配置文件一致性保证 | P1 |
| K8s 集群接入 | 接入已有 K8s 集群 | P0 |
| K8s 微服务部署 | Deployment、Service、Ingress | P0 |
| K8s 扩缩容 | HPA、手动扩缩容 | P1 |
| K8s 监控集成 | Prometheus 指标接入 | P1 |

### 关键里程碑

```
里程碑 2.1: 集群管理框架
  - 集群定义数据模型
  - SSH 批量操作工具封装
  - 节点添加/移除功能
  - 完成时间: 3 周

里程碑 2.2: K8s 集群接入
  - Kubeconfig 管理
  - 集群健康检查
  - Node/Pod/Service 列表查询
  - 完成时间: 2 周

里程碑 2.3: K8s 部署能力
  - YAML/Helm 部署
  - 多副本高可用配置
  - Pod 反亲和性配置
  - 完成时间: 4 周

里程碑 2.4: K8s 运维能力
  - 扩缩容操作
  - 日志/事件查看
  - Prometheus 监控集成
  - 完成时间: 3 周
```

### 预期成果

- 可管理 10+ 节点的集群
- K8s 微服务部署成功率 > 90%
- 支持主流中间件的高可用部署（MySQL、Redis、RabbitMQ）

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 2 人 | 需要 K8s 经验 |
| K8s 测试集群 | 1 套 | 3 节点以上 |
| 集群服务器 | 3-5 台 | 用于集群功能测试 |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 多节点操作一致性问题 | 高 | 高 | 引入 Ansible/Salt 作为编排层，保证幂等性 |
| K8s 权限控制复杂 | 中 | 中 | 使用 RBAC 最小权限原则，隔离不同租户操作 |
| 网络分区导致集群脑裂 | 低 | 高 | 设置操作锁，重大操作前检查集群健康状态 |

## 远期规划 (Phase 3)

### 目标
实现 K8s 自动化搭建，完善 DevOps 流程。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| K8s 集群自动搭建 | 基于 kubeasz | P1 |
| 拓扑规划模块 | 网络架构 + 部署架构规划 | P1 |
| 服务高可用配置 | 多副本 + PDB + 反亲和 | P0 |
| CI/CD 模板库 | 预定义发布流程模板 | P1 |
| GitOps 集成 | ArgoCD/Jenkins X 集成 | P2 |
| 自定义 Pipeline | 用户自定义流水线 | P2 |

### 关键里程碑

```
里程碑 3.1: K8s 自动搭建
  - kubeasz 工具封装
  - 配置文件生成
  - 搭建进度跟踪
  - 失败回滚机制
  - 完成时间: 4 周

里程碑 3.2: 拓扑规划
  - 用户需求解析
  - 节点角色分配建议
  - 网络架构建议
  - 完成时间: 3 周

里程碑 3.3: 服务高可用
  - StatefulSet 支持
  - 多副本 + PDB
  - 故障转移验证
  - 完成时间: 3 周

里程碑 3.4: CI/CD 基础
  - 预定义模板库
  - 模板选择向导
  - 基本 CI/CD 流程
  - 完成时间: 4 周

里程碑 3.5: DevOps 增强
  - GitOps 集成
  - 自定义 Pipeline
  - 完整 DevOps 流程
  - 完成时间: 6 周
```

### 预期成果

- 一键搭建生产级 K8s 集群（3-5 节点）
- AI 辅助的拓扑规划建议
- 完整的 DevOps 流程覆盖 80% 场景

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 2-3 人 | 需要 K8s + DevOps 经验 |
| K8s 搭建环境 | 5+ 节点 | 用于测试 K8s 搭建流程 |
| CI/CD 系统 | 1 套 | Jenkins/ArgoCD |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| K8s 搭建不可恢复失败 | 中 | 高 | 使用虚拟机/容器环境测试，充分验证回滚流程 |
| CI/CD 配置错误导致服务中断 | 中 | 高 | 蓝绿部署策略，保持旧版本可用 |
| AI 生成错误拓扑建议 | 中 | 中 | 提供拓扑预览，人工确认后再执行 |
| 自定义 Pipeline 语法复杂 | 高 | 低 | 提供可视化编辑器 + 模板市场，降低门槛 |

## 技术架构演进

### Phase 1 架构（当前）

```
用户输入 → Agent (LangGraph) → 工具注册表 → Docker/System 操作
                      ↓
                   CLI 输出
```

### Phase 2 架构

```
用户输入 → Agent (LangGraph) → 工具注册表
                      ↓
           ┌─────────┴─────────┐
           ↓                   ↓
      单节点操作           集群操作
      (Docker/System)     (SSH/Ansible)
                              ↓
                        K8s API (kubectl/helm)
```

### Phase 3 架构

```
用户输入 → Agent (LangGraph) → 工具注册表
                      ↓
           ┌─────────┴─────────┐
           ↓                   ↓
      运维操作            DevOps 流程
                              ↓
                    ┌─────────┴─────────┐
                    ↓                   ↓
               CI/CD 模板           GitOps
                    ↓                   ↓
              Jenkins/ArgoCD       Git Repo
```

```

10. 部署回滚能力

P2 (可以延后):
11. GitOps 集成
12. 自定义 Pipeline
13. 蓝绿部署策略
```

## 附录：名词解释

| 术语 | 解释 |
|------|------|
| Human-in-the-Loop | 关键操作需要人工确认后再执行 |
| 幂等性 | 同一操作执行多次结果一致 |
| PDB (PodDisruptionBudget) | 保证最小可用 Pod 数量 |
| 反亲和性 | 避免 Pod 调度到同一节点 |
| HPA (HorizontalPodAutoscaler) | Pod 水平自动扩缩容 |
| GitOps | 基于 Git 声明式配置的基础设施/应用交付 |
| Blue-Green Deployment | 蓝绿部署，新旧版本并行 |
