# Pulao 功能实现计划

## 项目概述

Pulao 是一个基于 LangGraph 的智能运维 Agent，通过自然语言完成 Docker 部署、系统运维、K8s 集群管理及 DevOps 流程编排。

**核心功能模块：**

1. 单节点服务器运维
2. 创建服务器集群
3. 基于服务器集群进行 K8s 运维（接入已有环境）
4. DevOps 流程

---

## 阶段划分总览

| 阶段 | 功能范围 | 优先级 | 建议周期 |
|------|----------|--------|----------|
| 近期 (Phase 1) | 单节点运维增强 | P0 | 1-2 个月 |
| 中期 (Phase 2) | 集群管理 + K8s 运维接入 | P1 | 2-3 个月 |
| 远期 (Phase 3) | DevOps 与 GitOps 集成 | P2 | 3-6 个月 |

## 近期规划 (Phase 1)

### 目标
完善单节点运维能力，稳定现有 Agent 架构，建立安全操作机制。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 单节点 Docker 运维 | 软件安装、服务部署、Docker Compose 部署 | P0 |
| 运维监控 | 日志查看、资源监控、故障诊断 | P0 |
| 安全确认机制 | 危险操作人工确认 (Human-in-the-Loop) | P0 |
| 操作回滚 | 部署失败自动回滚 | P1 |
| 模板库管理 | Docker Compose 模板的增删改查 | P1 |

### 关键里程碑

```
里程碑 1.1: 完成危险操作确认机制
  - 定义危险操作列表（rm, drop, shutdown 等）
  - 实现操作前置确认流程
  - 完成时间: 1 周

里程碑 1.2: 实现部署回滚能力
  - 备份当前部署状态
  - 部署失败自动触发回滚
  - 完成时间: 2 周

里程碑 1.3: 完善模板库
  - 模板的 CRUD 操作
  - 模板分类和搜索
  - 完成时间: 2 周

里程碑 1.4: 增强监控诊断
  - 日志分析
  - 资源告警
  - 完成时间: 2 周
```

### 预期成果

- 单节点运维场景可投入使用
- 危险操作 100% 经过人工确认
- 部署成功率 > 95%，失败回滚率 100%

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 1-2 人 | 兼职/全职 |
| 测试服务器 | 1-2 台 | 用于测试部署操作 |
| AI API 额度 | 足够测试使用 | DeepSeek/OpenAI |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| Agent 误判执行危险操作 | 中 | 高 | 强制 Human-in-the-Loop，关键操作设多级确认 |
| AI API 调用失败 | 低 | 中 | 实现重试机制，优雅降级为手动模式 |
| Docker 操作导致系统不稳定 | 中 | 中 | 测试环境先行验证，非破坏性操作优先 |

## 中期规划 (Phase 2)

### 目标
扩展到多节点集群管理，实现基于现有 K8s 环境的诊断与 Helm 部署能力。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 集群定义与管理 | 多台服务器的逻辑分组 | P0 |
| SSH 批量操作 | 基于 Ansible/Salt 保证执行幂等性 | P0 |
| 集群状态同步 | 配置文件一致性保证 | P1 |
| K8s 集群接入 | 通过 Kubeconfig 安全接入已有 K8s 集群 | P0 |
| K8s 诊断与查询 | 只读模式：查询 Pod/Service 状态、事件、日志 | P0 |
| K8s 声明式部署 | 通过生成 Helm Values 或原生 YAML 交由系统执行 | P0 |
| K8s 监控集成 | Prometheus 指标接入，基于指标提供扩缩容建议 | P1 |

### 关键里程碑

```
里程碑 2.1: 集群管理框架与一致性
  - 集群定义数据模型
  - 引入 Ansible 或类声明式引擎替换纯 SSH 循环
  - 节点添加/移除与状态预检
  - 完成时间: 3 周

里程碑 2.2: K8s 集群接入与诊断 (只读)
  - Kubeconfig 安全管理
  - 集群健康状态诊断
  - Node/Pod/Service 列表与事件查询
  - 完成时间: 2 周

里程碑 2.3: K8s 声明式部署能力
  - 限制 AI 直接操作 kubectl
  - 支持 AI 组装 Helm Values 进行部署
  - 多副本与反亲和性配置生成
  - 状态观察节点 (Observe)：验证部署结果
  - 完成时间: 4 周

里程碑 2.4: K8s 监控与建议
  - Prometheus 告警与指标接入
  - 结合指标生成 HPA 扩缩容配置建议（人工确认执行）
  - 完成时间: 3 周
```

### 预期成果

- 可稳定管理 10+ 节点的集群（保证幂等性）
- 能够诊断并排查现有 K8s 集群的问题
- 通过 Helm/YAML 安全、规范地部署高可用中间件

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 2 人 | 需要 K8s 运维经验 |
| 现有 K8s 集群 | 1 套 | 3 节点以上（用户提供） |
| 集群服务器 | 3-5 台 | 用于测试 Ansible/SSH 集群功能 |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 多节点操作一致性问题 | 中 | 高 | 必须引入 Ansible/Salt 等编排工具，放弃纯 Python SSH 循环 |
| AI 生成错误 K8s 配置引发故障 | 高 | 高 | **禁止命令式部署**，仅允许 AI 生成 YAML/Helm 配置，经系统校验并人工确认后执行 |
| K8s 权限控制风险 | 中 | 中 | 使用 RBAC 最小权限原则，初期仅授予只读权限和特定 Namespace 的部署权限 |

## 远期规划 (Phase 3)

### 目标
完善 DevOps 流程，实现声明式的 CI/CD 与 GitOps 闭环。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 服务高可用配置生成 | AI 辅助生成多副本、PDB、反亲和性配置 | P0 |
| 拓扑与架构建议 | 针对服务提供架构评估报告 | P1 |
| CI/CD 模板库 | 预定义常见项目的 CI/CD 发布流程模板 | P1 |
| GitOps 集成 | ArgoCD 集成：AI 生成配置直接推送到 Git | P1 |
| 自动化发布流水线 | 用户可通过对话生成并触发自动化流水线 | P2 |

### 关键里程碑

```
里程碑 3.1: 架构诊断与辅助配置
  - 对现有架构的自动巡检与拓扑建议
  - StatefulSet/Deployment 的基础配置生成
  - 完成时间: 3 周

里程碑 3.2: CI/CD 模板库
  - 预定义模板库建立
  - AI 根据项目语言智能推荐流水线
  - 完成时间: 3 周

里程碑 3.3: GitOps 闭环
  - AI 操作转化为 Git Commit (而不是直接部署)
  - 接入 ArgoCD 等工具自动同步配置
  - 完成时间: 4 周
```

### 预期成果

- AI Agent 完全融入 GitOps 标准工作流
- 提供专业的架构和部署建议
- 实现应用从代码到上线的自动化闭环

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 2-3 人 | 需要 CI/CD 与 GitOps 经验 |
| ArgoCD 测试环境 | 1 套 | 用于测试 GitOps 闭环 |
| CI 测试系统 | 1 套 | Jenkins/GitLab CI |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| AI 生成错误拓扑建议 | 中 | 中 | 所有建议需生成可视化报告并由人工复核 |
| 流水线定制语法复杂 | 高 | 低 | 限制 AI 仅使用内置的标准化模板库进行编排 |

## 技术架构演进

### Phase 1 架构（当前）

```
用户输入 → Agent (LangGraph) → 工具注册表 → Docker/System 操作
                      ↓
                   CLI 输出
```

### Phase 2 架构

```
用户输入 → Agent (LangGraph) → 工具注册表
                      ↓
           ┌─────────┴─────────┐
           ↓                   ↓
      单节点操作           集群操作
      (Docker/System)     (Ansible/SSH)
                              ↓
                        K8s 集群 (只读查询/Helm 部署)
```

### Phase 3 架构

```
用户输入 → Agent (LangGraph) → 工具注册表
                      ↓
           ┌─────────┴─────────┐
           ↓                   ↓
      运维操作            DevOps 流程
                              ↓
                    ┌─────────┴─────────┐
                    ↓                   ↓
               CI/CD 模板           GitOps
                    ↓                   ↓
               流水线触发           Git Repo -> ArgoCD -> K8s
```

## 优先级排序建议

根据实现复杂度和业务价值，建议优先级：

```
P0 (必须实现):
1. 单节点 Docker 运维（已实现，需增强如回滚机制）
2. Human-in-the-Loop 安全确认（已实现）
3. 集群操作一致性（引入 Ansible 替代裸 SSH）
4. K8s 集群接入与只读诊断
5. 基于 Helm/YAML 的 K8s 声明式部署

P1 (应该实现):
6. 拓扑与架构建议
7. 服务高可用配置生成
8. CI/CD 模板库
9. GitOps 闭环 (推送到 Git)

P2 (可以延后):
10. 自动化流水线触发
11. 结合 Prometheus 的智能扩缩容建议
```

## 附录：名词解释

| 术语 | 解释 |
|------|------|
| Human-in-the-Loop | 关键操作需要人工确认后再执行 |
| 幂等性 | 同一操作执行多次结果一致 |
| Helm Values | 用于配置 Helm Chart 部署的变量文件 |
| PDB (PodDisruptionBudget) | 保证最小可用 Pod 数量 |
| 反亲和性 | 避免 Pod 调度到同一节点 |
| HPA (HorizontalPodAutoscaler) | Pod 水平自动扩缩容 |
| GitOps | 基于 Git 声明式配置的基础设施/应用交付 |
| Blue-Green Deployment | 蓝绿部署，新旧版本并行 |
