# Pulao 功能实现计划

## 项目概述

Pulao 是一个基于 LangGraph 的智能运维 Agent，通过自然语言完成 Docker 部署、系统运维、K8s 集群管理及 DevOps 流程编排。

**核心功能模块：**

1. 单节点服务器运维
2. 创建服务器集群
3. 基于服务器集群进行 K8s 运维（接入已有环境）
4. DevOps 流程

---

## AI Agent 角色定位

### 核心定位：Copilot for DevOps

Pulao 定位为 **"运维工程师的智能助手"**，而非 **"自主运维系统"**。

```
┌─────────────────────────────────────────────────────────────┐
│                      人类运维工程师                          │
│                    (决策者 + 最终确认者)                      │
└─────────────────────────────────────────────────────────────┘
                              │
                              │ 提供自然语言指令
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                     Pulao AI Agent                          │
│                    (执行助手 + 顾问)                          │
│                                                             │
│  ✅ 擅长：                                                   │
│     - 将自然语言翻译为可执行的操作                            │
│     - 快速查询和聚合系统状态                                  │
│     - 生成配置建议和诊断报告                                  │
│     - 执行重复性、模板化的任务                                │
│                                                             │
│  ❌ 不擅长：                                                  │
│     - 承担生产事故的最终责任                                  │
│     - 做出需要业务上下文的架构决策                            │
│     - 处理 AI 未见过的新型故障模式                           │
└─────────────────────────────────────────────────────────────┘
```

### 角色定义

| 角色 | 描述 | 示例 |
|------|------|------|
| **翻译者** | 将自然语言意图翻译为具体操作指令 | "部署高可用 Nginx" → Helm Values |
| **诊断师** | 分析日志、指标，给出问题根因和修复建议 | 分析 OOMKilled 原因并建议调整内存限制 |
| **执行者** | 在人类确认后，执行预定义的安全操作 | 确认后执行 helm install |
| **顾问** | 基于最佳实践，提供架构和配置建议 | 建议配置反亲和性以提升可用性 |

## 核心场景定义

Pulao 聚焦于以下 3 个核心场景，而非追求大而全：

### 场景 A：日常运维的"快捷键"

**用户意图**：快速定位和解决常见问题

```
用户: "帮我看看 web-server 容器为什么重启了"
Agent:
  1. 查询容器状态 → exited with code 137 (OOMKilled)
  2. 分析内存使用趋势 → 发现内存峰值 2GB，限制 1.5GB
  3. 建议: 将内存限制调整为 2.5GB，或排查内存泄漏
  4. 置信度: 0.85
```

**价值**：减少运维工程师查日志、看指标的时间

### 场景 B：标准化部署的"模板生成器"

**用户意图**：快速生成符合规范的部署配置

```
用户: "帮我部署一个 3 节点的 Redis 集群，带持久化"
Agent:
  1. 生成 Helm Values (replicas: 3, persistence.enabled: true)
  2. 展示配置预览和资源影响评估
  3. 执行 --dry-run 验证
  4. 用户确认后执行 helm install
```

**价值**：降低标准化部署的门槛，减少手写 YAML 的错误

### 场景 C：故障诊断的"辅助医生"

**用户意图**：系统化排查复杂问题

```
用户: "服务响应变慢了，帮我排查"
Agent:
  1. 检查 Pod 状态 → 正常
  2. 检查 CPU/内存 → CPU 使用率 85%
  3. 检查网络延迟 → 正常
  4. 检查数据库连接池 → 发现连接数达到上限
  5. 建议: 增加连接池大小或检查是否有慢查询
  6. 置信度: 0.78
```

**价值**：系统化排查，避免遗漏关键检查点

## 边界与限制

### 明确不做的事情

| 不应该 | 原因 |
|--------|------|
| **自动执行危险操作** | 即使 AI 99% 准确，1% 的错误在生产环境也是灾难 |
| **替代监控系统** | Agent 是响应式的，不能替代 Prometheus/Grafana 的持续监控 |
| **处理复杂架构决策** | 架构决策需要业务上下文，AI 缺乏这些信息 |
| **承诺 100% 自动化** | 这会误导用户，导致过度信任 |

### 安全边界

| 操作类型 | 处理方式 |
|----------|----------|
| 只读查询（logs, status, describe） | 自动执行，无需确认 |
| 修改操作（deploy, scale, restart） | Dry-Run 预览 + 人工确认 |
| 危险操作（delete, rm, drop） | 多级确认 + 审计日志 |
| 不可逆操作（数据删除） | 强制备份 + 最终确认 |

## 近期规划 (Phase 1)

### 目标
完善单节点运维能力，稳定现有 Agent 架构，建立安全操作机制。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 单节点 Docker 运维 | 软件安装、服务部署、Docker Compose 部署 | P0 |
| 运维监控 | 日志查看、资源监控、故障诊断 | P0 |
| 安全确认机制 | 危险操作人工确认 (Human-in-the-Loop) | P0 |
| 操作回滚 | 部署失败自动回滚 | P1 |
| 模板库管理 | Docker Compose 模板的增删改查 | P1 |
| 置信度机制 | 工具调用附带置信度，低置信度强制确认 | P0 |
| 审计日志 | 所有操作记录到不可篡改的日志 | P0 |
| Dry-Run 模式 | 修改性操作默认预览，确认后执行 | P0 |

### 关键里程碑

```
里程碑 1.1: 完成危险操作确认机制
  - 定义危险操作列表（rm, drop, shutdown 等）
  - 支持用户自定义危险操作列表（配置文件扩展）
  - 实现操作前置确认流程（支持跳过/确认/取消）
  - 完成时间: 1.5-2 周

里程碑 1.2: 实现部署回滚能力
  - 设计部署状态快照机制（记录镜像版本、环境变量、挂载点）
  - 实现部署前自动备份
  - 部署失败自动触发回滚
  - 回滚操作日志记录
  - 完成时间: 2-3 周

里程碑 1.3: 完善模板库
  - 模板的 CRUD 操作
  - 模板分类和搜索
  - 模板版本管理
  - 完成时间: 2 周

里程碑 1.4: 增强监控诊断
  - 日志分析（实现滑动窗口截断，避免 LLM 上下文溢出）
  - 资源告警阈值配置
  - 故障诊断经验沉淀到 ChromaDB
  - 完成时间: 3 周

里程碑 1.5: 置信度机制与审计日志 (新增)
  - 工具调用时附带置信度评分 (0-1)
  - 置信度 < 0.8 时强制人工确认，并展示替代方案
  - 操作审计日志（不可篡改，包含时间戳、用户、意图、操作、结果）
  - 审计日志存储路径: ~/.pulao/audit.log
  - 完成时间: 1.5 周

里程碑 1.6: Dry-Run First 模式 (新增)
  - 所有修改性操作默认生成执行计划但不执行
  - 展示影响评估（资源变化、潜在风险）
  - 用户显式确认后才执行
  - 完成时间: 1 周
```

### 预期成果

- 单节点运维场景可投入使用
- 危险操作 100% 经过人工确认
- 部署成功率 > 95%，失败回滚率 100%
- Agent 响应延迟 P95 < 30s
- 所有操作可审计追溯
- 低置信度操作强制人工介入

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 1-2 人 | 兼职/全职 |
| 测试服务器 | 1-2 台 | 用于测试部署操作 |
| AI API 额度 | 足够测试使用 | DeepSeek/OpenAI |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| Agent 误判执行危险操作 | 中 | 高 | 强制 Human-in-the-Loop，关键操作设多级确认 |
| AI API 调用失败 | 低 | 中 | 实现重试机制，优雅降级为手动模式 |
| Docker 操作导致系统不稳定 | 中 | 中 | 测试环境先行验证，非破坏性操作优先 |

## 中期规划 (Phase 2)

### 目标
扩展到多节点集群管理，实现基于现有 K8s 环境的诊断与 Helm 部署能力。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 集群定义与管理 | 多台服务器的逻辑分组 | P0 |
| SSH 批量操作 | 基于 Ansible/Salt 保证执行幂等性 | P0 |
| 集群状态同步 | 配置文件一致性保证 | P1 |
| K8s 集群接入 | 通过 Kubeconfig 安全接入已有 K8s 集群 | P0 |
| K8s 诊断与查询 | 只读模式：查询 Pod/Service 状态、事件、日志 | P0 |
| K8s 声明式部署 | 通过生成 Helm Values 或原生 YAML 交由系统执行 | P0 |
| K8s 监控集成 | Prometheus 指标接入，基于指标提供扩缩容建议 | P1 |

### 关键里程碑

```
里程碑 2.1: 集群管理框架与一致性
  - 集群定义数据模型
  - Ansible 集成（使用 ansible-runner Python 库，非 subprocess 调用）
  - 目标节点 Python 环境预检（无 Python 时降级为 SSH 模块执行）
  - 幂等性测试用例设计
  - 节点添加/移除与状态预检
  - 完成时间: 3-4 周

里程碑 2.2: K8s 集群接入与诊断 (只读)
  - Kubeconfig 安全管理（支持多集群切换）
  - 集群健康状态诊断
  - Node/Pod/Service 列表与事件查询
  - 完成时间: 2 周

里程碑 2.3: K8s 声明式部署能力
  - 限制 AI 直接操作 kubectl（通过工具层封装）
  - 支持 AI 组装 Helm Values 进行部署
  - Helm Values Schema 校验（基于 values.schema.json）
  - 部署前 diff 预览（helm diff 或 template 对比）
  - --dry-run 自动化验证
  - 多副本与反亲和性配置生成
  - 状态观察节点 (Observe)：验证部署结果
  - 完成时间: 4-5 周

里程碑 2.4: K8s 监控与建议
  - Prometheus 告警与指标接入
  - 结合指标生成 HPA 扩缩容配置建议（人工确认执行）
  - 完成时间: 3 周
```

### 预期成果

- 可稳定管理 10+ 节点的集群（保证幂等性）
- 能够诊断并排查现有 K8s 集群的问题
- 通过 Helm/YAML 安全、规范地部署高可用中间件

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 2 人 | 需要 K8s 运维经验 |
| 现有 K8s 集群 | 1 套 | 3 节点以上（用户提供） |
| 集群服务器 | 3-5 台 | 用于测试 Ansible/SSH 集群功能 |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 多节点操作一致性问题 | 中 | 高 | 必须引入 Ansible/Salt 等编排工具，放弃纯 Python SSH 循环 |
| AI 生成错误 K8s 配置引发故障 | 高 | 高 | **禁止命令式部署**，仅允许 AI 生成 YAML/Helm 配置，经系统校验并人工确认后执行 |
| K8s 权限控制风险 | 中 | 中 | 使用 RBAC 最小权限原则，初期仅授予只读权限和特定 Namespace 的部署权限 |

## 远期规划 (Phase 3)

### 目标
完善 DevOps 流程，实现声明式的 CI/CD 与 GitOps 闭环。

### 功能清单

| 功能 | 描述 | 优先级 |
|------|------|--------|
| 服务高可用配置生成 | AI 辅助生成多副本、PDB、反亲和性配置 | P0 |
| 拓扑与架构建议 | 针对服务提供架构评估报告 | P1 |
| CI/CD 模板库 | 预定义常见项目的 CI/CD 发布流程模板 | P1 |
| GitOps 集成 | ArgoCD 集成：AI 生成配置直接推送到 Git | P1 |
| 自动化发布流水线 | 用户可通过对话生成并触发自动化流水线 | P2 |

### 关键里程碑

```
里程碑 3.1: 架构诊断与辅助配置
  - 对现有架构的自动巡检与拓扑建议
  - StatefulSet/Deployment 的基础配置生成
  - 配置生成后的 yamllint 校验
  - 高危挂载路径拦截（禁止挂载 /etc、/、/root 等）
  - 完成时间: 3 周

里程碑 3.2: CI/CD 模板库
  - 预定义模板库建立（支持 Jenkins/GitLab CI/GitHub Actions）
  - AI 根据项目语言智能推荐流水线
  - 完成时间: 3 周

里程碑 3.3: GitOps 闭环
  - AI 操作转化为 Git Commit (而不是直接部署)
  - 接入 ArgoCD 等工具自动同步配置
  - Git Revert 自动化流程（一键回滚到上一个版本）
  - ArgoCD Rollback 集成
  - 同步失败自动告警与回滚建议
  - 完成时间: 4-5 周
```

### 预期成果

- AI Agent 完全融入 GitOps 标准工作流
- 提供专业的架构和部署建议
- 实现应用从代码到上线的自动化闭环

### 资源需求

| 资源 | 数量 | 说明 |
|------|------|------|
| 开发人员 | 2-3 人 | 需要 CI/CD 与 GitOps 经验 |
| ArgoCD 测试环境 | 1 套 | 用于测试 GitOps 闭环 |
| CI 测试系统 | 1 套 | Jenkins/GitLab CI |

### 潜在风险与应对

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| AI 生成错误拓扑建议 | 中 | 中 | 所有建议需生成可视化报告并由人工复核 |
| 流水线定制语法复杂 | 高 | 低 | 限制 AI 仅使用内置的标准化模板库进行编排 |

## 附录：名词解释

| 术语 | 解释 |
|------|------|
| Human-in-the-Loop | 关键操作需要人工确认后再执行 |
| 幂等性 | 同一操作执行多次结果一致 |
| Helm Values | 用于配置 Helm Chart 部署的变量文件 |
| PDB (PodDisruptionBudget) | 保证最小可用 Pod 数量 |
| 反亲和性 | 避免 Pod 调度到同一节点 |
| HPA (HorizontalPodAutoscaler) | Pod 水平自动扩缩容 |
| GitOps | 基于 Git 声明式配置的基础设施/应用交付 |

## 技术决策记录 (ADR)

| 决策点 | 选型 | 原因 | 替代方案 |
|--------|------|------|----------|
| Agent 框架 | LangGraph | 原生支持循环与状态管理，解决复杂多步任务的死循环问题 | LangChain Agent (已弃用) |
| 集群编排 | Ansible + ansible-runner | 成熟、幂等、无需 Agent，Python 库集成方便 | SaltStack (更复杂) |
| K8s 部署 | Helm | 生态丰富，支持 Values Schema 校验 | Kustomize (功能较弱) |
| GitOps | ArgoCD | 声明式、可视化、社区活跃 | Flux (学习曲线更陡) |
| 向量数据库 | ChromaDB | 轻量级本地存储，无需部署云端服务 | Pinecone (需要网络) |
| CLI 框架 | Typer + Rich | 现代化、类型安全、美观输出 | Click (功能较弱) |

## 设计原则

基于以上分析，Pulao 的核心设计原则总结如下：

### 1. 安全第一 (Security First)

```
原则: 永远不要让 AI 自动执行不可逆操作

实现:
├── Human-in-the-Loop: 所有修改性操作需要人工确认
├── 置信度机制: 低置信度 (< 0.8) 强制确认
├── Dry-Run First: 默认预览，显式确认后执行
├── 审计日志: 所有操作可追溯
└── 禁止直接 kubectl: 通过工具层封装，声明式部署
```

### 2. 声明式优于命令式 (Declarative over Imperative)

```
原则: AI 生成配置，系统执行配置

✅ 正确做法:
  AI 生成 Helm Values → Schema 校验 → Dry-Run → 人工确认 → Helm Install

❌ 错误做法:
  AI 直接执行 kubectl apply/delete/exec
```

### 3. 预期管理 (Expectation Management)

```
原则: 让用户知道 AI 是助手，不是全能的神

实现:
├── 展示置信度，让用户了解 AI 的确定程度
├── 提供替代方案，而非单一答案
├── 明确告知 AI 的局限性
└── 关键决策由人类最终拍板
```

### 4. 渐进式增强 (Progressive Enhancement)

```
原则: 先做好核心场景，再扩展边界

Phase 1: 单节点运维 (可控风险)
    ↓
Phase 2: 集群 + K8s (引入 Ansible 保证幂等性)
    ↓
Phase 3: GitOps (融入标准工作流)
```

### 5. 可观测性 (Observability)

```
原则: Agent 的每一步都应该是透明可追溯的

实现:
├── 执行计划预览
├── 影响评估
├── 审计日志
├── 错误原因解释
└── 回滚能力
```
